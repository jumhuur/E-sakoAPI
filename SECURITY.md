# Security Policy

Thank you for helping keep EsakoAPI safe. If you discover a security issue, please report it responsibly so it can be reviewed and fixed before public disclosure.

## Supported Versions

Security fixes are handled for the latest version of the project on the default branch.

| Version | Supported |
| --- | --- |
| Latest `main` branch | Yes |
| Older commits or forks | No |

## Reporting a Vulnerability

Please do not open a public GitHub issue for security vulnerabilities.

Instead, report the issue privately by email:

```text
jumhuur123@hotmail.com
```

Please include as much detail as possible:

- A clear description of the vulnerability
- Steps to reproduce the issue
- Affected endpoint, file, or feature
- Potential impact
- Any logs, screenshots, proof of concept, or request examples
- Suggested fix, if you have one

## What to Avoid

When testing or reporting security issues, please do not:

- Access, modify, delete, or exfiltrate data that does not belong to you
- Disrupt the live service
- Run denial-of-service tests
- Publish exploit details before the issue is fixed
- Use social engineering, phishing, or credential attacks

## Response Process

After a report is received:

1. The issue will be reviewed and reproduced where possible.
2. If confirmed, a fix will be prepared.
3. The reporter may be contacted for clarification.
4. Public disclosure should wait until the fix is released.

This is a volunteer open-source project, so response times may vary. Serious reports will be prioritized.

## Scope

Security reports may include:

- Authentication or authorization issues, if such features are added
- Data exposure
- Injection vulnerabilities
- Unsafe file access
- Dependency vulnerabilities
- Misconfigured CORS or headers
- Issues that could affect API availability or integrity

Reports about Zakat calculation correctness are important, but they are not security vulnerabilities. Please report those through normal GitHub issues or pull requests.

## Disclosure

Please give the maintainer reasonable time to investigate and fix confirmed vulnerabilities before sharing details publicly.

Responsible disclosure helps protect users and keeps the project healthy.
